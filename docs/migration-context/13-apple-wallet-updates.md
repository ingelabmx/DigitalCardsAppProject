# Apple Wallet Updates

This phase adds the Apple Wallet Web Service required for installed passes to
update after stamps change.

## Runtime Behavior

Apple passes generated by the modern app now include:

- `webServiceURL`: `{DigitalCards:PublicBaseUrl}/apple-wallet`
- `authenticationToken`: deterministic HMAC token derived from
  `DigitalCards:AppleWallet:AuthenticationTokenSecret`

The token itself is never stored. The app stores a SHA-256 hash in the new
`AppleWalletPass` table.

## Web Service

The ASP.NET Core app exposes:

```text
POST   /apple-wallet/v1/devices/{deviceLibraryIdentifier}/registrations/{passTypeIdentifier}/{serialNumber}
DELETE /apple-wallet/v1/devices/{deviceLibraryIdentifier}/registrations/{passTypeIdentifier}/{serialNumber}
GET    /apple-wallet/v1/devices/{deviceLibraryIdentifier}/registrations/{passTypeIdentifier}?passesUpdatedSince=...
GET    /apple-wallet/v1/passes/{passTypeIdentifier}/{serialNumber}
POST   /apple-wallet/v1/log
```

Registration, unregister and pass download require:

```text
Authorization: ApplePass {authenticationToken}
```

The list endpoint returns `204 No Content` when no passes changed.

## HostGator Tables

Run this script manually after backing up the database:

```text
docs/migration-context/13-apple-wallet-updates-hostgator.sql
```

It creates new tables only:

- `AppleWalletPass`
- `AppleWalletDevice`
- `AppleWalletRegistration`

It does not modify `ApplePass`, `ClientCard`, Web Forms tables or stored
procedures.

## Push Updates

When a business adds a stamp:

- `ClientCard` is updated.
- Google Wallet is patched when already issued.
- Apple pass update tag is advanced when the pass was installed/downloaded.
- APNs update pushes are sent to registered devices for that pass.

APNs uses the configured Pass Type ID certificate `.p12`, with
`DigitalCards:AppleWallet:ApnsBaseUrl` defaulting to `https://api.push.apple.com`.
The payload is `{}` as required for pass update notifications.

## iPhone Smoke

Use an HTTPS public tunnel and keep `DigitalCards:PublicBaseUrl` stable during
the test.

1. Start the app with MySQL, Google, SMTP and Apple providers real.
2. Register a controlled test client.
3. Enroll that client with a controlled test business.
4. Open the received email on iPhone.
5. Select Apple Wallet and install the pass.
6. Confirm a row appears in `AppleWalletDevice` and `AppleWalletRegistration`.
7. Add a stamp from the business screen.
8. Confirm APNs returns success and Wallet fetches the updated `.pkpass`.

Do not log or commit `.p12`, passwords, push tokens, auth tokens, service account
JSON, SMTP credentials or generated `.pkpass` files.
